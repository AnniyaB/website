V0.1 Documentation

Your project documentation should contain following sections:

  • Team page: As before. Use the same template. Change '[V0.0]' to '[V0.1]' in the page header.
  • Credits: Acknowledgements of any third party materials used (if any). You may also list any extra (i.e. not required by the module) tool/technique you used (we want to know who went beyond the given set of tools/techniques). Omit this section if you did not use any third party materials.
  • User guide: As before. You may modify this section if you’ve changed the features since the last submission. It is ok if some of the features mentioned here are not implemented yet.
  • Developer guide: The intended audience for this document is an incoming developer (i.e. a new project member) that will depend on this document to tell how the product has been designed and implemented.That is, it will be used for onboarding new developers. The document contains just enough information required for that person to get into the project. The basic questions you should ask yourself when writing this document are:
    • What do we know about the system at this point?
    • Which part of that knowledge will be useful for a new project member?
    • What is the best way to pass that knowledge to the new member? Note that some information can be passed on in the form of code without any additional documentation.

    Given below are some things you could consider including. Note that you are not required to include them all. Neither you are restricted to what is listed below.

    • Architecture
    • important APIs of each component
    • Design descriptions (can include class diagrams, sequence diagrams, notable algorithms ...)
    • Code examples (only if necessary to explain an intricate implementation detail)
    • Instructions for setting up and testing

    Suggested max length for this section: 10 pages excluding cover page and appendices.

submission: Submit a single pdf file that contain all sections described above. File name is as before, but change ‘V0.0’ to ‘V0.1’. Submit to the appropriate IVLE folder. You may use GitHub to create an online user guide and a developer guide (see the collate project for an example), but you should still submit the project manual as a pdf file.
